# 如何设计一个秒杀系统？⭐

## 📌 题目分析

**秒杀场景特点：**
- 瞬时流量极大（万级 QPS → 百万级 QPS）
- 库存有限（供不应求）
- 时间集中（通常在整点）
- 读多写少（浏览多，下单少）

**核心挑战：**
1. 如何抗住高并发？
2. 如何防止超卖？
3. 如何保证公平性？
4. 如何防止作弊？

## 💡 核心解决方案

### 1. 流量削峰 - 多层过滤

```
用户请求 → CDN → Nginx 限流 → 网关限流 → Redis 预扣减 → Kafka 排队 → 数据库
          ↓        ↓           ↓           ↓           ↓          ↓
        拦截     拦截       拦截       拦截       异步      持久化
       90%      50%        80%        90%       缓冲      写入
```

**各层过滤比例：**
| 层级 | 过滤策略 | 预期拦截率 |
|------|---------|-----------|
| CDN | 静态资源缓存 | 90% |
| Nginx | IP 限流、黑名单 | 50% |
| 网关 | 用户限流、令牌桶 | 80% |
| Redis | 库存预扣减 | 90% |
| Kafka | 消息队列缓冲 | 100% 缓冲 |

### 2. 防止超卖 - 库存扣减方案

#### 方案一：Redis 原子操作（推荐）

```java
// Lua 脚本保证原子性
String luaScript = """
    local stock = tonumber(redis.call('GET', KEYS[1]))
    if stock <= 0 then
        return 0
    end
    redis.call('DECR', KEYS[1])
    return 1
""";

Long result = redisTemplate.execute(
    new DefaultRedisScript<>(luaScript, Long.class),
    Collections.singletonList("seckill:stock:" + productId)
);

if (result == 1) {
    // 扣减成功，发送消息到队列
    kafkaTemplate.send("seckill-order", orderId);
    return Result.success();
} else {
    return Result.fail("库存不足");
}
```

**优点：**
- 性能极高（Redis 单线程原子操作）
- 天然防超卖

**缺点：**
- Redis 宕机可能丢失数据
- 需要数据库最终对账

#### 方案二：数据库乐观锁

```sql
-- 使用版本号控制
UPDATE product 
SET stock = stock - 1, version = version + 1 
WHERE id = #{productId} 
  AND stock > 0 
  AND version = #{version}
```

```java
// Java 代码实现
int updateCount = productMapper.decreaseStock(productId, version);
if (updateCount > 0) {
    // 扣减成功
    return Result.success();
} else {
    return Result.fail("库存不足或版本冲突");
}
```

**优点：**
- 数据一致性强
- 实现简单

**缺点：**
- 性能较差（数据库压力大）
- 大量失败重试

### 3. 限流策略

#### 令牌桶算法实现

```java
@Component
public class RateLimiter {
    private final RateLimiter limiter = RateLimiter.create(1000); // 1000 QPS
    
    public boolean tryAcquire() {
        return limiter.tryAcquire();
    }
}

// Controller 使用
@RestController
public class SeckillController {
    @Autowired
    private RateLimiter rateLimiter;
    
    @PostMapping("/seckill/{productId}")
    public Result seckill(@PathVariable Long productId) {
        if (!rateLimiter.tryAcquire()) {
            return Result.fail("排队人数过多，请稍后再试");
        }
        // 处理秒杀逻辑
    }
}
```

#### 用户维度限流

```java
// 限制每个用户只能请求 N 次
String key = "seckill:limit:" + userId + ":" + productId;
Long count = redisTemplate.opsForValue().increment(key);
if (count == 1) {
    redisTemplate.expire(key, 60, TimeUnit.SECONDS);
}
if (count > 5) {
    return Result.fail("操作过于频繁");
}
```

### 4. 防作弊方案

| 手段 | 实现方式 | 效果 |
|------|---------|------|
| **登录验证** | 必须登录才能参与 | 基础门槛 |
| **答题验证** | 秒杀前答简单问题 | 增加脚本成本 |
| **IP 限流** | 单 IP 限制请求次数 | 防止 IP 攻击 |
| **设备指纹** | 识别设备唯一性 | 防止设备刷单 |
| **风控系统** | 用户行为分析 | 识别异常用户 |
| **黑名单** | 记录作弊用户 | 永久封禁 |

## 🔧 完整流程

```
1. 用户访问秒杀页面
   ↓ (CDN 缓存静态页面)
2. 用户点击"立即抢购"
   ↓ (网关限流、用户资格校验)
3. Redis 预扣减库存
   ↓ (Lua 脚本原子操作)
4. 扣减成功 → 发送消息到 Kafka
   ↓ (异步处理)
5. 消费者创建订单
   ↓ (数据库持久化)
6. 返回用户排队中
   ↓ (前端轮询结果)
7. 订单创建完成 → 通知用户支付
```

## ⚠️ 常见问题及解决方案

### Q1: Redis 宕机怎么办？

**方案：**
1. Redis Cluster 高可用
2. 库存数据持久化到数据库
3. 定时对账，恢复库存

### Q2: 用户支付超时怎么办？

**方案：**
1. 订单设置过期时间（15 分钟）
2. 延迟队列检测超时订单
3. 超时自动取消，释放库存

### Q3: 如何保证一人一单？

**方案：**
1. 数据库唯一索引约束
2. Redis Set 记录已购买用户
3. 下单前先检查资格

```java
// Redis 检查
Boolean isNewUser = redisTemplate.opsForSet()
    .add("seckill:users:" + productId, userId);
if (!isNewUser) {
    return Result.fail("每人限购一单");
}
```
